# hypercube-engine

Математически корректный движок рендера n-мерного гиперкуба (n до 12)
с независимыми перспективными проекциями на каждом уровне и
управляемой n-D камерой.

## Запуск

```bash
npm install
npm run dev          # dev-сервер vite на http://localhost:5173
npm run typecheck    # проверка TS без сборки
npm run test         # vitest (math-инварианты)
npm run build        # production-сборка в dist/
```

Размерность задаётся через URL hash: `index.html#n=10`. По умолчанию `n=4`.

## Математическая модель

### Вершины и комбинаторика

Гиперкуб `{−1, +1}ⁿ` представлен **строго через битовые операции**.
Вершина — целое число `v ∈ [0, 2ⁿ)`, где бит `i` числа `v` задаёт
знак `i`-й координаты:

    coord_i(v) = (v >> i) & 1 ? +1 : −1

Это даёт точную биекцию между подмножествами `{0, …, n−1}` и вершинами
куба — основа для теоретико-множественных представлений.

Прочие комбинаторные объекты:

* **Ребро** `(a, b, axis)`: `a < b`, `a ⊕ b = 2^axis`. Рёбра
  партиционированы по оси (`hypercube.edgesByAxis[i]`).
* **`k`-грань** задаётся парой `(freeMask, fixedValues)`:
  битовая маска свободных осей (popcount = k) + значения остальных
  координат. Количество `k`-граней: `C(n, k) · 2^(n−k)`.
* **Соседи**: `hypercube.neighbors[v][i] = v ⊕ (1 << i)`.

Никаких float-сравнений в структурных запросах — геометрия и топология
гиперкуба полностью сохраняются.

### n-D камера

Состояние камеры:

* `position ∈ ℝⁿ` — точка обзора в мире.
* `frame ∈ O(n)` — ортонормальный базис, строки которого суть
  базисные векторы камеры в мировых координатах.

Преобразование «мир → камера»:

    v_cam = frame · (v_world − position)

### Повороты Гивенса (инкрементальные)

Поворот в плоскости `(i, j)` на дельта-угол:

    frame ← G_{i,j}(dθ) · frame

Это вращает строки `i` и `j` матрицы `frame` в связке за O(n) операций.
В UI 45 «pad-контроллеров» (для `n=10`) выдают потоки малых `dθ` —
никаких абсолютных углов, никаких проблем с порядком композиции.

После каждых ~200 вращений или при достижении ошибки ортогональности
выше `1e−6` (норма Фробениуса `‖MMᵀ − I‖`) автоматически вызывается
модифицированный Грам-Шмидт, возвращающий матрицу на многообразие `O(n)`.

### Рекурсивная перспективная проекция

`ℝⁿ → ℝ³` строится как композиция `n − 3` шагов, каждый со
**своей** фокусной дистанцией `d_k`:

Шаг `d → d−1` (для `d` от `n` до `4`):

    w = v[d−1]                  // «глубокая» координата
    margin = d_focal + w
    if margin ≤ ε:  → CLIPPED   // точка позади камеры на этом уровне
    k = d_focal / margin
    v[0..d−2] *= k
    отбрасываем v[d−1]

При `d_focal = +∞` шаг становится ортогональным (`k = 1`, просто сброс
координаты). Это даёт UX «сборки куба»: при поэтапном уменьшении
`d_i` от `∞` к конечному значению пользователь видит, как очередное
измерение «прорастает» из предыдущего.

Важное замечание: при identity-ориентации камеры ортогональная проекция
коллапсирует структуру. Стартовая поза — каскад `G_{i,i+1}(π/9)`,
протекающий высшие оси в видимые координаты (см. `Camera.setReadablePose`).

### Индексация дистанций

`projection.distances[k] = d_{k+4}`:

* `distances[0]` — фокус для уровня `4 → 3` (`d_4`)
* `distances[1]` — для `5 → 4` (`d_5`)
* …
* `distances[n−4]` — для `n → n−1` (`d_n`)

API: `setDistance(sourceDim, value)`, `getDistance(sourceDim)`. Значение
`+Infinity` допустимо и означает ортогональную проекцию.

### Клиппинг рёбер

Ребро `[V1, V2]` проецируется покаскадно. На каждом уровне:

* если оба конца впереди (`margin > ε`) — обычная проекция;
* если оба позади — ребро отбрасывается;
* иначе ищется `t ∈ (0, 1)` пересечения с гиперплоскостью
  `w = −d_focal + ε`, «плохой» конец двигается в эту точку,
  затем обе точки проецируются.

Это приближение: отрезок треков «реальной» проективной кривой
линейно в редуцированной системе на каждом уровне. Для рёбер 1-куба
{−1, +1}ⁿ погрешность мала; для критичных случаев можно добавить
субдивизию в `Projection.projectEdge`.

«Event horizon»: при `minMargin < threshold` (по умолчанию `15%` от
минимального `d_i`) цвет ребра линейно интерполируется к ярко-красному
`(1, 0.1, 0.05)` — предупреждение перед сингулярностью.

### Knife-clip culling

Отбраковка 2-граней в n-D. Грань с `n−2` фиксированными координатами
`x_a = s_a` видна снаружи, если камера лежит в «outward-октанте»:

    ∀ фиксированной оси a:   C[a] · s_a > 1

При камере внутри куба (`∀ i: |C[i]| < 1`) критерий не проходит ни одна
грань — в этом режиме knife-clip автоматически отключается, и
работает только 3D back-face culling Three.js поверх.

## Использование математического ядра без рендера

Модули `math/`, `culling/`, `color/` не зависят от `three`.
Их можно импортировать в любой Node/браузер контекст:

```typescript
import { Hypercube } from './src/math/Hypercube.js';
import { Camera } from './src/math/Camera.js';
import { Projection } from './src/math/Projection.js';

const h = new Hypercube(10);
console.log(h.vertexCount, h.edgeCount);   // 1024, 5120

// Подмножество вершин, лежащих в 3-ячейке с x_0 = x_1 = +1:
const cell = h.buildFace({ 0: 1, 1: 1, 3: -1, 4: 1, 5: 1, 6: -1, 7: 1 });
const verts = h.faceVertices(cell);        // 8 вершин (2^3)

// Спроецировать вершину через камеру:
const cam = new Camera(10);
cam.setReadablePose();
const proj = new Projection(10);
const worldCoords = h.coords(0);
const camCoords = new Float64Array(10);
cam.transformWorldToCamera(worldCoords, camCoords);
const result = proj.projectVertex(camCoords);
console.log(result.pos);  // [x, y, z] в ℝ³
```

## UI

* **Depth Stack** — слайдеры `d_i` для каждого уровня проекции +
  чекбокс «ortho» (`d_i = ∞`). Наведение на слайдер подсвечивает
  соответствующую группу рёбер.
* **Plane Selector** — `n × n` решётка, верхне-треугольная часть
  кликабельна. Выбирает активную плоскость вращения. По умолчанию
  `(0, 1)`.
* **Rotation Pad** — drag-область. Горизонтальное движение мышью →
  инкрементальное вращение в активной плоскости.
* **Highlights** — кнопка `show 3-cell` выделяет каноническую
  3-ячейку (свободные оси `0, 1, 2`, остальные `+1`). Чекбоксы для
  knife-clip и event-horizon-предупреждения.
* **Diagnostics** — ошибка ортогональности, число вершин/рёбер/квадов.

Three.js камера для финального 3D → 2D шага управляется обычными
`OrbitControls`: вращение колесом мыши, зум скроллом. Это
**не** влияет на n-D проекцию — она про другую геометрию.
